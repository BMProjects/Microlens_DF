# 自动化迭代研发准备（2026-03-24）

## 1. `autoresearch` 方法对当前项目的直接启发

参考 `karpathy/autoresearch` 官方 README，其最值得迁移到本项目的不是“代理自动改代码”本身，而是四个实验组织原则：

1. 单一可控编辑面
2. 固定训练预算
3. 固定主指标
4. 明确的保留/丢弃规则

对当前项目，这四点应改写成：

- 识别算法分支与分割算法分支分开推进
- 每轮实验只改变一个主要变量
- 识别分支固定看 `CNAS` 留出测试集
- 分割分支固定看 `mIoU / scratch IoU / mask->bbox AP@0.5`
- 不能证明优于父实验的方案不进入正式主线
- `CNAS` 20 张留出图像只作为最终核验，不再参与应用系统和后端算法实验改进的日常决策

## 2. 识别算法分支准备

当前识别主线结论：

- 正式基线仍为 `stage2_cleaned`
- `B2_nwd_only` 是唯一已在固定留出测试集上略优于 baseline 的结构改进
- `SFE-only` 与 `SFE+NWD` 当前结果退化
- `SSGD` 迁移学习当前未证明有效

因此识别分支下一阶段的自动化迭代顺序应为：

1. `NWD-only` 复现实验作为父实验
2. 在固定预算下做单变量消融
3. 每次都在 `CNAS` 固定测试集上复评
4. 仅保留 `scratch AP@0.5` 提升且 `mAP@0.5` 不低于 baseline 的实验

## 3. 分割算法分支准备

分割分支对正式系统影响较小，适合作为独立研发线加速推进。当前仓库已具备：

- `LightUNet` 基线
- `MSD` 预训练数据准备
- 私有弱标签生成
- 全图滑窗推理与 `mask_to_detection_result()`

本轮进一步准备已完成：

- 训练框架支持固定预算训练
- 分割模型工厂支持多模型扩展
- 第一批可直接对照的模型已收敛为：
  - `light_unet`
  - `unetplusplus`
  - `deeplabv3plus`
  - `fpn`
- 新增批量实验入口：`scripts/run_segmentation_experiments.py`

## 4. 为什么先推进这四个分割模型

这四个模型的选择依据是“在当前依赖和工程复杂度下可落地”，并与研究报告的方向相衔接：

- `light_unet`：私有弱标签训练最稳定的轻量基线
- `unetplusplus`：适合验证多尺度跳连对细长结构的收益
- `deeplabv3plus`：适合验证大感受野与边界恢复
- `fpn`：适合验证更轻的多尺度融合语义分割

更进一步的 `SegFormer / HRNet-OCR / Mask2Former` 建议放在下一轮，引入专门框架后再推进。

## 5. 当前建议的推进顺序

### 5.1 识别算法

1. 固化 `baseline` 与 `NWD-only`
2. 建立按单变量改动的研究记录
3. 所有实验统一走 `CNAS` 子系统评测

### 5.2 分割算法

1. 在 `MSD` 数据集上跑四个结构基线，作为源域结构排名
2. 选择最优两个结构进入私有弱标签微调
3. 私有数据集按“训练验证数据集”口径使用，不将其表述为测试集
4. 对比 `mIoU / scratch IoU / mask->bbox AP@0.5`
5. 再决定是否引入边界损失、clDice 或更强 backbone
