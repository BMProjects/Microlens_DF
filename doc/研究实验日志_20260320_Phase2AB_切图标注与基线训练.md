### Exp-P2A-01：全图检测内存崩溃分析

**问题描述**：
- 第 1 张图（1l.bmp）：检出 529 个缺陷实例，耗时约 5 分钟
- 第 2 张图开始处理后：`exit code 144`（OOM kill，内存溢出）

**根因分析（定量）**：

```python
# 内存占用分析
DefectInstance.mask = np.ndarray(shape=(3000, 4096), dtype=bool)
内存占用 = 3000 × 4096 × 1 byte = 12.3 MB / 实例

# 场景一：第 1 张图
529 实例 × 12.3 MB = 6.5 GB → 超出物理内存上限

# 场景二：划痕合并算法复杂度
划痕片段数量 n ≈ 290（第 1 张图）
O(n²) 配对比较 = 290 × 290 = 84,100 次（实际有效对约 42,000）
→ CPU 占用峰值，与内存同时超负荷
```

**方案对比（全图 vs 切图）**：

| 对比维度 | 全图 4096×3000 | 切图 640×640 |
|----------|--------------|-------------|
| 单实例 mask 内存 | 12.3 MB | 0.41 MB（缩小 **30 倍**）|
| 100 实例总 mask | 1.23 GB | 41 MB |
| 划痕合并复杂度 | n≈290，O(n²)≈42,000 | n≈20，O(n²)≈190（减少 **221 倍**）|
| ML 训练适配性 | 需额外切割 | 直接可用 |
| 单次推理速度 | ~5 min | ~0.1 s/tile |

**关键设计决策**：切图处理彻底解决内存问题，并将输出格式与 YOLO 训练直接对接，一步到位。

### Exp-P2A-02：图像增强策略对标注质量的影响——4 版本迭代实验

**研究问题**：哪种图像增强策略能在最小化噪声放大的同时最大化缺陷检出率？

**增强链路设计**：

```
原始切片（640×640 灰度）
    │
    ├── [亮度通道]  在原始图上运行 → 高亮 spot/damage 检测
    │
    └── [增强分支]  gamma → CLAHE → Frangi（线状）+ TopHat（亮斑）
                   → 候选图 → Otsu 阈值 → 形态学 → 实例分类
```

**各版本详细对比（测试集：10 张图像，约 430 tiles）**：

**版本 v1（原始方案，gamma=0.4）**：

```
检出统计：
  scratch:  8,657    spot:  2,467    crash:  1,027    damage: 15
  总计 tiles 平均：28.3 缺陷/tile

问题1：高亮大缺陷漏检
  Frangi 滤波器为脊线检测器，对实心亮块响应极低
  TopHat 核 = 15 px，大于 15 px 的亮区域 TopHat 响应 = 0
  实测：原始图中灰度>30 且面积>100 px² 的区域有 139 个，大部分未被检出

问题2：噪声放大
  gamma = 0.4 的变换：f(x) = 255 × (x/255)^0.4
  灰度值 3（噪声底层）→ 43（14× 放大）
  ROI 内 91.6% 像素为噪声底层 → 均被提亮到可检出范围
```

**版本 v2（增加亮度通道，但在增强图上运行）**：

```
检出统计：
  scratch:  5,273    spot:  1,867    crash:  3,015
  总计 tiles 平均：23.6 缺陷/tile

新问题：crash 暴增 3 倍（3,015 vs 1,027）
  根因：brightness_channel 在 gamma 增强后的图像上运行
  增强图 gamma=0.4 后：21.9% 像素被判定为"高亮"（远超合理比例）
  高亮像素密集 → crash 密度检测误报爆炸
```

**版本 v3（亮度通道改在原始图上运行）**：

```
检出统计：
  scratch:  3,700    spot:  1,356    crash:  2,084
  总计 tiles 平均：16.6 缺陷/tile

改进：crash 数量合理化（比 v2 下降 31%）
残留问题：gamma=0.4 仍然放大噪声，spot 标注仍有噪声混入
```

**版本 v4（p90 基准 + gamma=0.55，最终采用方案）**：

```
检出统计：
  scratch:  3,858    spot:  825    crash:  835
  总计 tiles 平均：12.8 缺陷/tile ← 最优

关键修改 1：亮度通道基准从全图均值改为 ROI p90 百分位
  bg_floor = np.percentile(roi_pixels, 90)  # ≈ 3（原始图）
  resp = (img - bg_floor) / (p95 - bg_floor)
  效果：只有超过 90% 像素亮度的才被认定为高亮缺陷

关键修改 2：gamma 提升
  gamma: 0.4 → 0.55
  灰度 3：14× → 7×（噪声放大减半）
  灰度 20：4.6× → 3.3×（有效缺陷提亮略微减少，可接受）
```

**关键发现（写入设计约束）**：
亮度通道**必须**在原始图（非增强图）上运行。这是避免噪声误检爆炸的核心设计。

**最终采用参数**（`configs/detect_tile.yaml`）：

```yaml
enhance_gamma: 0.55
clahe_clip_limit: 3.0
otsu_floor: 0.08          # 防止纯暗 tile 阈值过低
min_area: 120             # 最小连通域面积（从全图模式的 50 上调，过滤噪声碎片）
min_length: 30.0          # 划痕最小长度（px，从 20 上调）
density_kernel_ratio: 0.15 # crash 密度检测核比例（tile 更小，核比例调大）
```

**参数调整依据（640px tile vs 全图模式对比）**：

| 参数 | 全图模式 | 切图模式 | 调整依据 |
|------|---------|---------|---------|
| min_area | 50 | 120 | tile 中噪声连通域更密集，提高以过滤 |
| min_length | 20 | 30 | 切片后划痕片段短，20px 几乎都是噪声 |
| density_kernel_ratio | 0.08 | 0.15 | tile 面积为全图 1/36，核比例需等比上调 |

### Exp-P2A-03：YOLO 三类训练方案设计

**背景**：经典检测器原始输出 4 类：scratch / spot / damage / crash

**类别不均衡分析**（247 张图、10,621 tiles 完整标注结果）：

| 类别 | 实例数 | 占比 | 风险评估 |
|------|--------|------|---------|
| scratch | 75,995 | 74.5% | 主导梯度，模型可能过拟合于划痕 |
| spot | 19,997 | 19.6% | 尚可，正常训练 |
| crash | 16,891 | 16.6% | 尚可，正常训练 |
| **damage** | **46** | **0.045%** | **极度稀缺，基础训练召回率接近 0** |
| 合计 | 112,929 | 100% | — |

**关键决策：damage + crash 合并为 critical（YOLO 三类方案）**

- 合并依据：damage（孤立大面积高亮，面积 > 2000 px²）与 crash（高密度缺陷聚集区）在视觉影响评估中均属于"密集/大面积严重缺损"，对镜片透光率的影响类型相似
- YOLO 训练使用 3 类：`scratch=0, spot=1, critical=2`
- **保留内部 4 类分类器**（`scoring/` 模块），用于 WearScore 磨损评分
- 合并后 critical 类：16,891 + 46 = 16,937 实例，占比提升到正常水平

**最终数据集规模**（`output/tile_dataset/`）：

| 指标 | 数值 |
|------|------|
| 源图像 | 247 张 |
| 总 tile 数 | 10,621 |
| train split | 8,471 tiles（80%）|
| val split | 2,150 tiles（20%）|
| 有缺陷 tile | 10,417（98.1%）|
| 空白负样本 tile | 204（1.9%）|
| 原始标注总量 | 112,929 个框 |

**数据集划分策略**：以原图为单位划分 train/val，保证同一镜片的所有切片不跨越 train/val 边界，避免数据泄露。

## Phase 2B：YOLOv12m 基线训练

### 模型选型

**YOLOv12m**（ultralytics 8.4.24，Area Attention 内置）

- 参数量：约 20M
- 输入通道：灰度图像复制为 3 通道（方案 A：gray×3）
- 损失函数：Varifocal Loss + DFL + Focal Loss（fl_gamma=2.0）

**输入通道方案对比**：

| 方案 | 输入构成 | 预期提升 | 代价 | 决策 |
|------|---------|---------|------|------|
| A: gray×3 | 灰度直接复制 3 通道 | 基线（0%）| 无额外成本 | **Stage1 使用** |
| B: [gray, frangi, clahe] | 三通道语义编码 | scratch +3-5% mAP | 预处理时间 ×2 | Stage2B 备选（未实施）|

方案 B 的优势：将经典检测器的领域知识注入输入通道——Frangi 通道显式编码线状结构，对 scratch（宽度仅 1-10 px）尤其有帮助。但时间约束优先完成基线路径。

### Exp-P2B-01：Stage1 基线训练设计与执行

**假设**：YOLOv12m 在半自动标注（含噪声）的 10,621 tiles 上，能达到 ≥ 35% mAP@0.5 基线性能。

**完整训练配置**（`scripts/train_stage1_baseline.py`）：

```yaml
model:          yolo12m.pt          # ImageNet 预训练权重
imgsz:          640
epochs:         100
batch:          16                  # RTX 4090D VRAM 24 GB
lr0:            0.01
lrf:            0.01                # 终止 LR = lr0 × lrf = 1e-4（余弦退火）
warmup_epochs:  3
fl_gamma:       2.0                 # Focal Loss，缓解 scratch/spot 类别不均衡
patience:       20                  # 早停轮数

# 暗场专用增强设置（禁用颜色增强，灰度图像无意义）
hsv_h:      0.0                     # 关闭色相增强
hsv_s:      0.0                     # 关闭饱和度增强
hsv_v:      0.4                     # 亮度扰动 ±40%（模拟散射强度变化）

flipud:     0.5

```

**预计训练时长估算**（RTX 4090D）：

```
8,471 tiles / batch 16 = 530 iter/epoch
~0.1 s/iter × 530 = 53 s/epoch × 100 epochs ≈ 88 分钟
```

**实际训练结果**：

| Epoch | mAP@0.5 | Precision | Recall | 备注 |
|-------|---------|-----------|--------|------|
| ep22 | **0.5257** | 0.5422 | 0.5047 | **最优点（早停基准）** |
| ep38 | 0.33 | — | — | 最终 epoch（早停触发）|

**训练过程问题诊断**：

| 指标 | 数值 | 解读 |
|------|------|------|
| train/val loss gap | **0.73** | 典型严重过拟合（正常应 < 0.1）|

### Exp-P2B-02：标注噪声深度分析——4 类系统性噪声识别

**研究问题**：半自动标注中存在哪些可量化、可程序化修复的系统性噪声？

**分析方法**：以 Stage1 推理结果审计标注质量，结合数据统计和人工目视确认。

**发现的 4 类系统性噪声**（`scripts/step1_label_cleanup.py`）：

| 噪声类型 | 数量 | 占原始标注比例 | 根因 | 检测规则 |
|----------|------|--------------|------|---------|
| **A. 噪声小框** | 2,972 | 2.63% | 经典检测器对局部亮度波动（相机传感器噪声）的假检 | 面积 < min_area_thresh |
| **B. 边缘截断残留** | 6,970 | 6.17% | 切片边界截断缺陷框，保留极小残片（宽或高 < 15 px 且边距 < 10 px）| 边距 < 10 px 且（宽 < 15 或 高 < 15）|
| **C. 满切片大框** | 1,829 | 1.62% | 密集划痕区域整张切片被框为单一大矩形（面积 > 切片面积 50%）| area_ratio > 0.50 |
| **D. 跨切片重复标注** | 7,140 | 6.32% | overlap=80 px 导致相邻切片重复标注同一缺陷 | IoU > 0.70 的重叠框对 |
| **合计** | **15,338** | **13.6%** | — | — |

**关键发现**：

**类型 C（满切片大框）** 是 Stage1 极低 val recall（9.85%）的主要原因：
- 密集划痕图像（整体亮度高，多条划痕密集交叉）中，经典检测器将整个切片内的连通域合并为一个覆盖全切片面积 50%+ 的大框
- YOLO 模型无法从"整个切片 = 一个 scratch"中学习有效的缺陷边界特征
- 这类标注提供了错误的空间先验，严重干扰梯度

**清洗执行结果**：

```bash
python scripts/step1_label_cleanup.py
# 执行结果：112,929 → 97,591 个标注（移除 13.6%）
```

### Exp-P2B-03：多策略推理伪标签生成实验

**研究问题**：如何利用 Stage1 模型补充经典检测器的漏标缺陷，提升 recall？

**方法设计**（`scripts/step2_pseudo_labels.py`）：

**多策略推理（6 组）**：

```
策略维度 1：推理尺度（图像 resize 大小）
  尺度 640 px：与训练分辨率一致，标准精度
  尺度 960 px：放大 1.5×，增强小缺陷（spot）检测
  尺度 1280 px：放大 2×，进一步细化

策略维度 2：图像增强
  原图（无 TTA）
  水平翻转（TTA）

6 组 = 3 尺度 × 2 增强
```

**伪标签筛选条件**：

```python
# 跨策略一致性筛选（保守策略，避免过多噪声）
consistent = (出现策略数 >= 2) AND (avg_confidence >= 0.35)

# 与 GT 对比（补漏逻辑）
is_new = (与所有 GT 框的 max_IoU < 0.5)
# 仅将 GT 中没有对应标注的预测框加入伪标签集
```

**工程问题及解决方案**：

```
问题：首次运行时 OOM
  原因：batch=32 × 8,471 tiles 一次性分配 GPU 显存
  解决：stream=True 推理模式 + 按尺度分批
    640 px → batch=64（显存充裕）
    960 px → batch=32
    1280 px → batch=16（显存紧张）
```

**执行结果**：

| 阶段 | 数量 | 说明 |
|------|------|------|
| 清洗后 GT 训练集 | 75,464 框 | Step1 清洗后的 train split |
| 多策略推理候选 | 21,455 框 | 6 组推理 + 跨策略一致性筛选 |
| 新增补漏标注 | **8,613 框** | 与 GT IoU < 0.5，视为真实漏标 |
| 融合后训练集 | **84,077 框** | 较清洗后 GT **+11.4%** |

**Stage1 → Stage2 对比（提升量）**：

### Exp-P2B-05：Frangi 验证辅助工具——标注质量审计

**工具**：`scripts/step1b_frangi_verify.py`

**方法**：对所有 GT scratch 框计算 Frangi 线状结构响应值，用响应强度量化标注可信度：

```python
frangi_response = frangi(tile_gray, sigmas=[1, 2, 3], black_ridges=False)
bbox_response = frangi_response[y1:y2, x1:x2].mean()
# 响应 < 0.05：无线状结构 → 噪声误标
# 响应 0.05-0.30：弱线状结构 → 可疑（边缘片段）
# 响应 > 0.30：明确线状结构 → 可信
```

**统计结果（train split）**：

| Frangi 响应范围 | 占比 | 解读 |
|----------------|------|------|
| < 0.05（无线状结构）| 44.3% | 噪声误标（可能是亮度波动被标为 scratch）|
| 0.05 ～ 0.30（弱响应）| 13.1% | 切片边缘碎片（划痕只露出极小片段）|
| > 0.30（明确线状）| 42.6% | 可信标注 |

**可疑框总量**：20,263 个（train split，占 train scratch 框约 57.4%）

**影响评估**：这是 scratch AP 低于 spot/critical 的主要结构性原因之一（见 Phase 2C 详细分析）。
