# 后续工作准备（2026-03-24）

## 1. 当前项目的真实主体功能与架构判断

### 1.1 当前“可对外展示”的主体其实已经是 GUI，而不是 CLI

从当前实现看，项目最完整、最接近可交付产品形态的入口是：

- `src/darkfield_defects/viz/app.py`

GUI 已经形成了一个完整工作台：

- 输入镜片图像
- 加载默认权重 `output/training/stage2_cleaned/weights/best.pt`
- 调用 `scripts/infer_full_pipeline.py` 中的全图推理能力
- 输出 4 个核心面板：
  - 检测叠加图
  - 缺陷地形图
  - WearScore 计算过程图
  - 评级卡片

这意味着后续“第一预发布版本”的主体应明确以 GUI 为核心，而不是继续把 CLI 视作主产品入口。

### 1.2 当前代码实际上存在“两条主线”

#### 主线 A：产品可视化主线

- GUI：`src/darkfield_defects/viz/app.py`
- 全图推理：`scripts/infer_full_pipeline.py`
- 检测模型：`output/training/stage2_cleaned/weights/best.pt`
- 推理逻辑：SAHI 风格切片推理 + 全图 IOS NMS + `connect_scratches`
- 评分逻辑：GUI 当前实际使用的是 `scripts/infer_full_pipeline.py` 中“微结构区优先”的评分版本

#### 主线 B：包内传统算法主线

- CLI：`src/darkfield_defects/cli/app.py`
- 预处理：`src/darkfield_defects/preprocessing/pipeline.py`
- 检测：`src/darkfield_defects/detection/classical.py`
- 评分：`src/darkfield_defects/scoring/quantify.py` + `src/darkfield_defects/scoring/wear_score.py`

这两条线目前并没有真正统一：

- GUI 走的是 YOLO/全图推理/微结构区优先评分
- CLI 走的是传统检测 + 包内 WearMetrics/WearScore

因此，当前最重要的架构任务不是继续加功能，而是先“收主线”。

### 1.3 当前第一预发布版本建议的功能边界

第一预发布版本建议只保留下面这条清晰主线：

1. 图像输入
2. 全图推理
3. 缺陷可视化
4. 评分与评级
5. 结果导出

建议暂时不把下列研究性能力放进预发布主线：

- SSGD 迁移学习实验
- SFE/NWD 结构改造实验
- 私有分割训练实验
- CNAS 第三方测试脚本
- 传统检测器单独作为主入口

### 1.4 第一预发布版本的建议架构

建议形成如下结构：

```text
GUI（唯一产品入口）
  -> app/service/inference_service.py
      -> preprocess_service.py
      -> detection_service.py
      -> scoring_service.py
      -> export_service.py

研究实验分支
  -> scripts/train_*.py
  -> scripts/prepare_*.py
  -> scripts/compare_*.py

独立测试分支
  -> cnas_test/
      -> manifests/
      -> templates/
      -> docs/
      -> runner/
      -> outputs/latest/
```

对应原则：

- GUI 只调服务层，不直接 import `scripts/`
- `scripts/` 保留为研究与批处理工具
- CNAS 测试逻辑独立，不和 GUI/研究脚本互相耦合

## 2. 前期实验的有效经验与当前改进结论

### 2.1 当前最稳的模型主线仍然是 `stage2_cleaned`

以固定留出测试集为准：

- baseline `mAP@0.5 = 0.6765`
- `scratch AP@0.5 = 0.4525`
- `spot AP@0.5 = 0.8180`
- `critical AP@0.5 = 0.7589`

当前这条主线是后续预发布版本应冻结使用的默认主线。

### 2.2 已被证明有效的改进

前期实验中，真正被验证有效的是：

1. 标注清洗
2. 伪标签补漏
3. 全图推理后的后处理合并
4. 固定留出测试集和固定 CNAS 评测参数

也就是说，当前提升主要来自：

- 数据质量提升
- 推理流程改造

而不是来自新网络结构本身。

### 2.3 已得到的关键负面经验

最重要的失败经验有三个：

1. `connect_scratches` 只能用于推理端，不能回写 GT
2. 任何重标注修改不能动 `val/test`
3. 新结构改进不能多个变量一起堆，必须做可比较消融

### 2.4 截至目前的模型改进结果判断

结合 `output/experiments/comparison/comparison_report.json`，当前判断如下：

- `A0_baseline`：当前正式基线
- `C1_ssgd_finetune`：总体几乎持平，没有体现稳定收益
- `C1_ssgd_freeze10`：失败
- `B1_sfe_only`：退化明显
- `B2_nwd_only`：当前唯一相对有希望的结构改进，`mAP50 = 0.6802`，略高于 baseline
- `B3_sfe_nwd`：退化明显

因此下一阶段不应平均推进所有路线，而应按证据优先级推进：

1. 先固化 baseline
2. 再以 `NWD-only` 为识别算法改进起点
3. 分割路线独立推进

## 4. 后续工作线 2：算法改进拆成两条研发分支

后续底层算法应明确拆成：

- 识别算法改进分支
- 分割算法分支

二者并行推进，但评估体系和产物必须独立。

### 4.1 识别算法改进分支

#### 当前起点

以 `stage2_cleaned` 为正式 baseline。

#### 当前最值得继续的方向

- `NWD-only`

理由：

- 已有固定测试集结果略优于 baseline
- 与 scratch 细长框 IoU 脆弱性问题直接对应
- 改造边界较清楚

#### 当前不建议优先投入的方向

- `SFE-only`
- `SFE+NWD`
- 直接继续堆 SSGD 迁移学习

这些路线目前要么退化，要么证据不足。

#### 参考 `karpathy/autoresearch` 的自动化迭代方式

建议把识别算法实验固化为自动循环：

```text
提出假设
  -> 自动生成实验配置
  -> 启动训练
  -> 固定留出测试集评测
  -> 自动汇总对比
  -> 产出下一轮假设
```

建议最小自动化单元为：

1. `experiment spec`：记录模型改动、训练参数、输入通道、损失函数
2. `runner`：统一训练入口
3. `evaluator`：固定调用留出测试集评测
4. `reporter`：自动生成实验对比表
5. `gatekeeper`：只接受超过 baseline 且无明显副作用的实验

建议后续新增：

```text
research/recognition/
  specs/
  runners/
  evaluators/
  reports/
```

#### 识别算法分支的迭代优先级

1. `NWD-only` 复验
2. 输入通道增强（例如 `[gray, frangi, clahe]`）
3. 仅在证据明确后再回到 SFE
4. 如继续做公开数据迁移学习，优先转为“教师模型辅助标注提质”而非直接提分

### 4.2 分割算法分支

#### 当前起点

当前仓库已经具备分割方向的基础设施：

- `scripts/generate_weak_masks.py`
- `scripts/prepare_msd_segmentation.py`
- `scripts/train_msd_segmentation.py`
- `scripts/train_private_segmentation.py`
- `src/darkfield_defects/ml/models.py`
- `src/darkfield_defects/ml/predict.py`

#### 分割分支的目标不应是直接替代检测

建议分三步走：

1. 先做“检测后精修”
2. 再做“分割辅助量化”
3. 最后才考虑“检测/分割联合主线”

#### 分割分支的关键评估指标

不能只看检测 mAP，应单独看：

- mask IoU
- pixel F1
- skeleton length error
- 区域面积误差
- 交叉点统计误差
- 最终评分稳定性

#### 分割分支的短期目标

优先形成：

- 可复现实验数据准备
- 固定训练脚本
- 固定评估脚本
- 1 个可用私有微调权重

## 6. 后续工作线 4：重建镜片评分理论与实现

### 6.1 当前评分实现存在的问题

当前项目里至少存在两套评分逻辑：

1. `src/darkfield_defects/scoring/wear_score.py`
2. `scripts/infer_full_pipeline.py` 中的评分逻辑

问题在于：

- 一套更偏“中心区/总量”逻辑
- 一套更偏“微结构区优先”逻辑
- 二者在理论上还没有统一成同一个评分体系

### 6.2 新评分理论的基本原则

离焦微结构镜片不是普通镜片，评分必须同时满足两点：

1. 中心区影响视觉清晰度，必须重点考虑
2. 微结构区承担离焦控制功能，属于相对重要功能区，不能被“总缺陷数”淹没

因此，后续建议保留现有五项主指标框架：

- `L_center`
- `L_total`
- `N`
- `A`
- `S_scatter`

但把它们全部改造成“功能区加权后的有效指标”。

### 6.3 建议的新评分定义

先定义三个区域：

- 中心区 `center`
- 微结构区 `micro`
- 边缘区 `edge`

再定义各项有效指标：

```text
L_total_eff = 1.0 * L_center + 0.8 * L_micro + 0.3 * L_edge
N_eff       = 1.5 * N_center + 1.2 * N_micro + 0.5 * N_edge + 1.5 * N_critical
A_eff       = 1.5 * A_center + 1.3 * A_micro + 0.5 * A_edge + 1.5 * A_critical
S_eff       = 1.4 * S_center + 1.2 * S_micro + 0.5 * S_edge
```

然后再将五项主指标统一映射到评分函数：

```text
Score =
  w1 * sat(L_center / T1) +
  w2 * sat(L_total_eff / T2) +
  w3 * sat(N_eff / T3) +
  w4 * sat(A_eff / T4) +
  w5 * sat(S_eff / T5)
```

其中：

- `sat(x)` 使用对数或指数饱和函数，避免极端单项把总分拉爆
- `T1~T5` 由训练验证数据集分位数标定

### 6.4 建议的权重逻辑

建议初版权重：

```text
w1 = 0.30   # 中心区划痕长度
w2 = 0.20   # 区域加权总长度
w3 = 0.15   # 区域加权缺陷数
w4 = 0.20   # 区域加权面积/严重缺损
w5 = 0.15   # 区域加权散射强度
```

理由：

- `L_center` 直接对应主视野干扰，单列最高
- `A_eff` 对大面积损伤和微结构区受损更敏感，应高于简单计数
- `S_eff` 反映夜间眩光和散射风险，不应再只是附属项
- `L_total_eff` 与 `N_eff` 作为全局损伤补充项

### 6.5 实现建议

评分重构建议分两步：

#### 第一步：统一指标层

在 `src/darkfield_defects/scoring/quantify.py` 中统一产出：

- `L_center`
- `L_micro`
- `L_edge`
- `N_center`
- `N_micro`
- `N_edge`
- `A_center`
- `A_micro`
- `A_edge`
- `S_center`
- `S_micro`
- `S_edge`
- `N_critical`
- `A_critical`

#### 第二步：统一评分层

在 `src/darkfield_defects/scoring/wear_score.py` 中实现唯一评分公式，并让 GUI 与 CLI 共同复用。

最终原则：

- 一个指标定义
- 一个评分公式
- 一套等级解释
- 一套导出结构

## 8. 当前建议的明确结论

1. 第一预发布版本应以 GUI 为唯一主体，冻结当前 `stage2_cleaned` 权重作为默认模型。
2. 研究工作不要继续混入主产品入口，应拆成“识别算法分支”和“分割算法分支”两条研发线。
3. CNAS 测试必须独立成子系统，和 GUI、训练、研究实验彻底解耦。
4. 评分系统必须统一到一套“中心区 + 微结构区优先”的正式理论中，并保留 `L_center/L_total/N/A/S_scatter` 五项主框架。
5. 当前最应该做的不是继续大规模换模型，而是先完成“主线收敛、评分统一、测试独立、实验自动化”这四件基础工作。
