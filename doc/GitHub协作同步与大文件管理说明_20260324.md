# GitHub 协作同步与大文件管理说明

## 1. 当前原则

当前项目准备同步到 GitHub 进行多人协作。为了保证同步效率、降低冲突概率，并避免将 `13G+` 的本地产物一并推送，仓库从现在开始采用以下原则：

1. `代码、配置、文档、测试、轻量级清单` 进入 Git
2. `数据集、切片、训练输出、实验结果、权重、审计图、发布压缩包` 默认不进入 Git
3. 文档中需要长期引用的插图，从 `output/` 提升到 `doc/assets/`

## 2. 当前建议纳入同步的内容

建议保留在 GitHub 仓库中的内容：

- `src/`
- `scripts/`
- `configs/`
- `tests/`
- `cnas_test/`
- `research_loops/`
- `doc/*.md`
- `doc/*.docx`
- `doc/assets/`
- `pyproject.toml`
- `uv.lock`
- `.gitignore`

这些内容体积相对可控，且直接影响：

- 系统可运行性
- 架构理解
- 团队协作
- 文档交接

## 3. 当前不建议同步的内容

以下内容建议排除在 GitHub 同步范围外。

### 3.1 数据集相关

- `output/dataset_v2/images/`
- `output/tile_dataset/images/`
- `output/tile_dataset/labels/`
- `output/tile_dataset/labels_backup_4class/`
- `output/tile_dataset/labels_backup_preclean/`
- `output/tile_dataset/overlays/`
- `output/release/microlens_df_dataset_v1.tar.gz`

原因：

- 体积大
- 变化频繁
- 容易造成网络同步缓慢
- 更适合放在本地磁盘、共享 NAS 或对象存储

### 3.2 训练与实验产物

- `output/training/`
- `output/experiments/`
- `runs/`
- `output/pseudo_labels/`
- `output/sahi_results/`

原因：

- 大量中间图、日志、曲线和权重
- 多人协作时几乎必然产生冲突
- 大多数产物可由脚本重新生成

### 3.3 权重与模型文件

- `*.pt`
- `*.pth`
- `yolo12m.pt`
- `yolo26n.pt`

原因：

- 体积较大
- 不适合跟随普通代码提交频繁变更
- 更适合独立发布或存放于共享模型仓

### 3.4 测试与审计运行产物

- `output/cnas_eval/`
- `output/audit/`

原因：

- 这些内容属于“执行结果”，不属于源码
- 需要保留的是执行逻辑和固定清单，不是每次运行产生的图片与 JSON

### 3.5 本地环境与缓存

- `.venv/`
- `__pycache__/`
- `.pytest_cache/`
- `.claude/settings.local.json`

## 5. 推荐的外部存储方式

GitHub 仓库只保留源码与轻量清单，大文件建议转为以下任一种：

1. 团队共享 NAS
2. 研发服务器固定目录
3. 对象存储 / 网盘
4. 独立数据仓库或 DVC / Git LFS（如后续确有需要）

当前阶段最实用的建议是：

- 原始数据与切片数据放共享盘
- 训练权重放共享模型目录
- GitHub 仓库只保留访问说明、清单和脚本

## 6. 当前目录归类建议

### 6.1 仓库内保留

```text
src/                    源码
scripts/                研究与批处理脚本
configs/                配置
tests/                  自动化测试
cnas_test/              独立测试子系统
doc/                    文档
doc/assets/             文档插图
research_loops/         自动化研发规则
```

### 6.2 仓库外或仅本地保留

```text
output/dataset_v2/      整图数据
output/tile_dataset/    切片数据与标签
output/training/        正式训练产物
output/experiments/     研究实验产物
output/audit/           审计结果
output/cnas_eval/       运行结果
runs/                   临时运行结果
*.pt / *.pth            模型权重
```

## 7. 建议的首次 GitHub 同步检查

首次推送前，建议按这个顺序检查：

1. 确认 `git status` 中没有 `output/` 下的大文件
2. 确认 `.venv/` 和 `runs/` 未被跟踪
3. 确认文档引用图片都来自 `doc/assets/`
4. 确认真正需要共享的清单文件已经迁到源码或文档目录

## 8. 结论

当前项目最适合的 GitHub 协作模式，不是把所有数据和结果都推上去，而是：

- 把仓库变成“源码 + 文档 + 轻量清单”的协作中心
- 把数据集、权重和运行产物留在本地或共享存储

这样既能保证多人协作效率，也能避免后续每次提交都被大文件拖慢。
